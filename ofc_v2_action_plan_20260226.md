## 3. 確定方針

上記を踏まえ、以下を確定方針とする。

### 方針 1: Config C を 5M ステップまで延長

```
目的: Config C の優位性が統計的に有意か確認する
所要時間: 約 4M ステップ追加 × 31分/1M ≈ 約 2時間
GCPコスト: e2-standard-8 × 2時間 ≈ 数百円程度
```

**5M到達時の判断基準:**

| 指標 | 採用条件（Config C続行） | 要再検討 |
|------|------------------------|----------|
| フォール率 | < 40%（かつA/Bより低い） | > 45% または A/B と逆転 |
| 平均スコア | > +4.0（かつA/Bより高い） | A/B との差が縮小 |
| FL Entry | > 1%（FL学習が始まっている） | 厳密には0%でも可（まだ早い場合あり） |
| FL追求フォール率差 | < 10% | > 15%（報酬ハッキングの強い兆候） |
| Explained Variance | > 0.15（改善傾向） | < 0.05（学習が停滞） |

**判定ロジック:**
- 上記の「採用条件」を概ね満たす → **Config C 正式採用、Phase 1 Step 2 へ進む**
- 「要再検討」に複数該当 → Config A との再比較、または報酬設計の再調整

### 方針 2: C++ エンジン網羅テストを並行実施

```
実施タイミング: Config C の 5M 学習をGCPで回している間に、ローカルMacで実装
目標: 学習完了前にテスト完了
```

**テストでカバーすべき領域（優先順）:**

| # | テスト対象 | 理由 | 優先度 |
|---|-----------|------|--------|
| 1 | ACE の扱い（A > K の確認） | V1崩壊の直接原因。修正済みだが網羅テストなし | 🔴 最優先 |
| 2 | ハンド比較演算（全役の大小関係） | Top < Mid < Bot のフォール判定に直結 | 🔴 最優先 |
| 3 | 役の境界判定（ストレート境界、A-2-3-4-5 vs 10-J-Q-K-A） | ローエンド/ハイエンドストレートの判定ミスはスコア直撃 | 🟠 高 |
| 4 | ジョーカーの扱い（全役での代用判定） | OFC Pineapple特有。Jokerが絡む役の正確性 | 🟠 高 |
| 5 | FL Entry 条件（QQ+判定、カード枚数） | FL学習の前提。ここが壊れていたら5M以降が全て無駄 | 🟠 高 |
| 6 | FL Stay 条件（Trips on Top + Mid≧Trips / Quads on Bot） | Phase 2以降で重要になる | 🟡 中 |
| 7 | スコア計算（3人 head-to-head、スクープ、ロイヤリティ） | 最終報酬の正確性 | 🟡 中 |
| 8 | フォール判定（Top < Mid < Bot の全パターン） | 基本中の基本だが網羅確認は必須 | 🟡 中 |

### 方針 3: 学習環境

```
当面: GCP（e2-standard-8）で学習継続
  - 報酬設定確定 + Step 2 初回比較まではGCPを維持

並行: ローカルMacで開発・テスト

切り替え検討タイミング:
  - Phase 2（30M学習）に入る前にコスト再評価
  - GCP Spot VM への移行も選択肢
```

## 6. 重要な注意事項（両スペシャリスト共通）

### やるべきこと

- **変数は1つずつ動かす** — 報酬設定を確定してからSelf-Play混合を変える。同時に動かすと因果関係が不明になる
- **C++ テストは学習と並行で必ず完了させる** — V1の二の舞を避ける最重要防衛線
- **FL Entry 0% に焦らない** — 1M〜5M は基礎安定化フェーズ。FLは自然に出始める
- **報酬ハック指標は5M以降に再評価** — 現時点ではサンプル不足で信頼性が低い

### やってはいけないこと

- **Config C を確定する前に Step 2 に進む** — 報酬とSelf-Playの2変数が混ざる
- **C++ テスト未完了のまま30M+ の本学習に入る** — V1で250M+が無駄になった教訓
- **FL Entry が 0% だからといって報酬を上げる** — 報酬ハッキングの再発リスク
- **GCP から Mac に急いで切り替える** — 安定比較が必要なフェーズで環境を変えない
