## 背景と動機

V1 プロジェクト（Phase 1〜12）の全モデルは、C++ evaluator の ACE=0 バグにより
**間違ったポーカールールで学習**していた。85M ステップの fine-tune でも矯正できなかったため、
修正済み evaluator（ACE=14）で**ゼロから**学習し直す。

さらに、外部スペシャリスト 2 名のレビューにより、V1 の学習設計には
**報酬ハッキング**、**行動空間の曖昧さ**、**評価基盤の不在** という構造的欠陥が判明。
V2 ではこれらを解消した上で学習を行う。

## 全体戦略（改訂版）

```
Phase 0          Phase 1             Phase 2            Phase 3           Phase 4
基盤整備    →    小規模検証     →    構造改善+中規模  →  高度化+本学習  →  実機テスト
(コードのみ)     (1M〜5M steps)      (5M〜30M steps)    (30M〜250M steps)  (評価)
```

### V1 のロードマップとの最大の違い

**V1**: いきなり学習開始 → 問題が後から発覚 → 手戻りの連続
**V2**: 基盤を先に整備 → 小規模で検証 → 問題がないことを確認してから本学習

### なぜこの順番か（ML 解説）

1. **まず道具を点検する**（Phase 0: テスト、評価、報酬の健全性確認）
2. **小さく試して問題を洗い出す**（Phase 1: 複数設定を 1M ステップずつ比較）
3. **構造を改善して中規模で確認**（Phase 2: 行動空間改善、30M ステップ）
4. **確信を持って本学習する**（Phase 3: 全改善を統合した最終版で 250M）
5. **実際のゲームで検証する**（Phase 4: mix-poker-app での対人テスト）

## Phase 0: 基盤整備（学習前）

> **目的**: 学習を始める前に、環境・報酬・評価系を健全な状態にする。
> V1 最大の失敗は「壊れた道具で長時間学習した」こと。同じ失敗を繰り返さない。

### 0-1. C++ エンジンの網羅テスト

ACE=14 修正が正しく動作するか、エッジケースを含めて検証する。

```
テスト対象:
  - ACE の kicker 比較（AA > KK > QQ > ... > 22）
  - Joker を含むハンド評価
  - FL Stay 条件（Trips on Top, Quads on Bottom）
  - フォール判定（Top < Mid < Bot の全組み合わせ）
  - ロイヤリティ計算（全ハンドランク）
```

**成果物**: `tests/test_evaluator_comprehensive.py`

### 0-2. 報酬ハッキングの修正

**問題**: FL 接近報酬の合計（+23）がフォールペナルティ（-6.2）を大幅に上回るため、
AI が「わざと Top を無理に強くして FL ボーナスを稼ぎ、フォールしても黒字」
という戦略を学習する可能性が高い。

**修正**: FL 接近報酬をフォールペナルティの 1/3 以下に縮小

| パラメータ | V1 の値 | V2 修正値 | 根拠 |
|:----------|-------:|--------:|:-----|
| fl_stage1_reward | 5.0 | **0.5** | フォール(-6.0) の 1/12 |
| fl_stage2_reward | 10.0 | **1.0** | フォール(-6.0) の 1/6 |
| fl_stage3_reward | 8.0 | **0.8** | フォール(-6.0) の 1/7.5 |
| **合計** | **23.0** | **2.3** | フォール(-6.0) の **1/3 以下** |

**ML 解説: 報酬ハッキング（Reward Hacking）とは？**
AI が「設計者の意図とは異なる方法で報酬を最大化する」現象。
人間なら「それはズルだ」と気づくが、AI は報酬を最大化するだけなのでズルもルール通りの行動も区別しない。
対策は「ズルが儲からないように報酬のバランスを調整する」こと。

### 0-4. 固定ベンチマーク相手の作成

学習中の AI の強さを客観的に測る仕組みを先に作る。

```
ベンチマーク相手:
  1. ランダム Agent — 下限ベースライン
  2. 安全型 Rule-Based — フォール回避最優先、FL は狙わない
  3. 攻撃型 Rule-Based — FL 積極追求、フォールリスク許容
  4. 過去チェックポイント — 1M, 5M, 10M 時点のモデルを固定

評価プロトコル:
  - 各相手と 1000 ゲーム以上
  - 全ポジション（BTN/SB/BB）で均等に
  - 統計的有意差の検定

評価指標（各対局で記録するもの）:
  - 勝率（%）— 勝ち/負け/引き分け
  - Net Score（平均得点差）— 勝率だけでは見えない差を補完
    ※ 勝率 60% でも Net Score が僅差なら実力差は小さい
    ※ 勝率 55% でも Net Score が大きければ実質的に強い
  - Foul Rate（%）— 相手タイプ別
  - FL Entry Rate（%）— 相手タイプ別
```

**Rule-Based Agent の二重役割**:
1. **ベンチマーク相手**（Phase 0 で作成）: 強さの客観評価用
2. **Self-Play 混合相手**（Phase 1 Step 2 で検証）: 学習中の対戦相手の多様性確保

Rule-Based Agent は Phase 0 で先に作成し、Phase 1 以降は
ベンチマーク評価と Self-Play 混合の両方に使い回す。

**成果物**: `v2/evaluate_benchmark.py`, `v2/rule_based_agent.py`

## Phase 1: 小規模検証学習（1M〜5M ステップ）

> **目的**: 複数の設定を小規模で比較し、Phase 2 以降の最良パラメータを決定する。
> V1 は「一発勝負で 250M ステップ」だったが、V2 は「1M で検証してから本番」。

### ML コンセプト: アブレーション実験（Ablation Study）とは？

「要素を1つずつ変えて効果を測る」実験手法。
薬の臨床試験で「薬あり群 vs プラセボ群」を比較するのと同じ発想。

> **重要: 実験の進め方**
> 全実験を一度に並列で走らせるのではなく、**3 ステップに分けて順番に行う**。
> 前のステップの結果を確認してから次に進むことで、実験の方向性を逐次修正できる。
> 全 7 本を同時に走らせると結果の解釈が困難になり、組み合わせ爆発を起こす。

### Step 1: 報酬アブレーション（最優先）

**なぜ最初か**: 報酬設計はモデルの学習方向そのものを決める。
ここが間違っていると他の設定をどう変えても無意味になるため、最初に確定させる。

3 種類の報酬設定で 1M ステップずつ比較:

| 設定 | FL 接近報酬 | FL Entry/Stay ボーナス | 目的 |
|:-----|:----------|:---------------------|:-----|
| **A. 最小** | Stage1=0.5, Stage2=1.0, Stage3=0.8 | Entry=30, Stay=60 | 報酬ハッキングが起きないか確認 |
| **B. ゼロ** | 全て 0 | Entry=50, Stay=100 | 中間報酬なしでも FL を学べるか |
| **C. 条件付き** | ゲーム終了時、非フォール時のみ付与 | Entry=30, Stay=60 | ポテンシャルベース報酬の有効性 |

**監視すべきこと**: 設定 A で「FL 追求時のフォール率」が通常時より高くないか。
高ければ報酬ハッキングが起きている。

**Step 1 完了後の判断**: A/B/C から最良の報酬設定を 1 つ選び、以降の全実験に適用。

### Step 2: Self-Play プール拡張の検証

**なぜ 2 番目か**: 対戦相手の構成は「誰から学ぶか」を決める。
報酬が確定した上で、対戦環境の最適化を行う。

Step 1 で選定した報酬設定を使い、以下を比較:

| 設定 | pool_size | latest_prob | rule_based | 目的 |
|:-----|:---------:|:-----------:|:----------:|:-----|
| **D. V1 設定** | 5 | 0.8 | 0% | ベースライン |
| **E. 拡張** | 15 | 0.6 | 10% | 偏り防止効果を確認 |

**Step 2 完了後の判断**: D/E からプール設定を選定。

### Step 3: 学習主体ローテーションの検証

**なぜ最後か**: 座順バイアスは「あると困るが、なくても学習自体は成立する」種類の問題。
報酬と対戦相手が確定してから検証することで、ローテーションの純粋な効果を測れる。

Step 1 + Step 2 で選定した設定をベースに:

| 設定 | 学習主体 | 目的 |
|:-----|:---------|:-----|
| **F. V1 設定** | player_0 固定 | ベースライン |
| **G. ローテーション** | 毎エピソードでランダム | 座順バイアスの除去 |

**Step 3 完了後の判断**: ポジション別スコア差が ±3 以上ならローテーション採用。

### Phase 1 の評価方法

各設定を 1M ステップ実行し、Phase 0 で作ったベンチマーク相手と対局。
評価指標は **勝率** に加え **Net Score（平均得点差）** も記録する。
勝率はスコア差を反映しないため、「僅差の勝ち」と「大差の勝ち」を区別する Net Score が補完指標として重要。

**最良設定の選定基準**:
1. 報酬ハッキングが発生していない（FL 追求時のフォール率 ≤ 通常フォール率 + 5%）
2. Foul Rate が最も低い
3. ベンチマーク相手への Net Score が最も高い
4. ベンチマーク相手への勝率が最も高い

### サーマルスロットリング計測

Intel Mac での長時間 CPU 負荷は、サーマルスロットリングにより後半の FPS が低下するリスクがある。
Phase 1 の実験中に以下を計測し、Phase 2 以降の学習計画（所要時間）を現実的に見積もる:

```
計測項目:
  - 学習開始時の FPS（最初の 10 分間平均）
  - 3 時間後の FPS
  - CPU 温度（可能なら powermetrics で記録）
  - 低下率（%）= (開始時 FPS - 3h後 FPS) / 開始時 FPS × 100

目安:
  - 低下率 < 10%: 問題なし。NUM_ENVS=8 を維持
  - 低下率 10-25%: NUM_ENVS を 6 に減らして再計測
  - 低下率 > 25%: NUM_ENVS=4 に制限し、タイムラインを 1.5 倍に修正
```

## Phase 2: 構造改善 + 中規模学習（5M〜30M ステップ）

> **目的**: Phase 1 で選定した最良設定に構造的改善を加え、30M ステップで基礎力を確立する。

### 2-1. 行動空間の分離（270 次元方式）

**問題**: 現在の Discrete(243) は Street 1（5枚配置, 3^5=243 通り）と
Street 2-5（2枚配置+1枚捨て, 27 通り）が同じ番号を共有している。
action=42 が Street 1 では「Top, Bot, Mid, Mid, Top」を意味するが、
Street 2-5 では全く別の操作を意味する。

**修正**: 270 次元に拡張し、フェーズで完全分離

```
Discrete(270):
  index 0-242:   Street 1 用（5枚配置, 3^5=243 通り）
  index 243-269: Street 2-5 用（2枚配置+1枚捨て, 27 通り）

マスクで制御:
  Street 1:   index 0-242 のみ有効、243-269 は全てマスク
  Street 2-5: index 0-242 は全てマスク、243-269 のみ有効
```

MaskablePPO のアクションマスクで完全に切り替えるため、
ネットワークの出力層が 27 次元分増えるだけで既存パイプラインとの互換性が高い。

必要に応じて「ポテンシャルベース報酬」（非フォール時のみ FL 接近報酬を付与）を実装。

### 成功基準

成功基準を **「健全性」** と **「強さ」** の 2 層に分離する。
健全性は「学習が正しく機能しているか」、強さは「どれだけ強いか」を測る。
**健全性が不合格なら、強さの数値がどれだけ良くても失敗とみなす**（V1 の教訓）。

#### Tier 1: 健全性基準（全項目必須。1 つでも不合格なら中断）

| 指標 | 基準 | 意味 |
|:-----|:-----|:-----|
| Foul Rate | < 30% | ルール理解の最低ライン |
| Explained Variance | > 0.3 | 価値関数が機能している |
| FL 追求時 Foul Rate | ≤ 通常 + 5% | 報酬ハッキングが起きていない |
| EV（期待値）の方向性 | 非フォール時 EV > フォール時 EV | ゲームの基本戦略を理解 |

#### Tier 2: 強さ基準（目標。未達でも Phase 3 で到達可能なら続行）

| 指標 | 基準 | 根拠 |
|:-----|:-----|:-----|
| FL Entry | > 5% | FL 学習の開始 |
| Mean Score | > +5 | スコアリングの理解 |
| Net Score vs 安全型 | > +2.0 | ベンチマーク基礎力 |
| ベンチマーク安全型に勝率 | > 60% | 基礎力の確認 |

### 中断基準
- 10M で Tier 1 健全性基準を 1 つでも未達かつ改善傾向なし → 報酬/ネットワーク見直し
- Explained Variance < 0.1 → 価値関数の問題を調査

## Phase 3: 高度化 + 本学習（30M〜250M ステップ）

> **目的**: 全改善を統合した最終版で長期学習を実施し、V1 Phase 9/10 を超える。

### 3-1. 学習率スケジュール

| ステップ範囲 | Learning Rate | ent_coef | 備考 |
|:------------|:-------------|:---------|:-----|
| 30M–80M | 3e-4 | 0.03 | Phase 2 から継続 |
| 80M–150M | 1e-4 | 0.02 | 微調整フェーズ |
| 150M–250M | 5e-5 | 0.015 | 最終収束フェーズ |

### 3-2. FL 報酬の段階的強化

80M ステップ以降、基礎力が安定したら FL ボーナスを段階的に増加:

| ステップ | FL Entry Bonus | FL Stay Bonus | 条件 |
|:--------|:---------------|:-------------|:-----|
| 30M–80M | +30 | +60 | 基礎学習の継続 |
| 80M–150M | +50 | +100 | Foul Rate < 25% を維持した場合のみ |
| 150M–250M | +50 | +100 | 維持 |

**重要**: FL ボーナス増加は Foul Rate が悪化しないことを確認してから行う。
Foul Rate が増加した場合はボーナスを元に戻す。

### 3-3. 定期ベンチマーク評価

10M ステップごとにベンチマーク相手と自動対局を実施し、記録する。

### 成功基準

Phase 2 と同じく **「健全性」** と **「強さ」** の 2 層構造。

## Phase 4: 実機テスト

> **目的**: 学習した AI を mix-poker-app に統合し、実際のゲームプレイで品質を検証する。
> V1 最大の反省は「数値は良いのに実際のプレイが悪かった」こと。

### やること

1. **ONNX エクスポート**: 学習済みモデルを ONNX 形式に変換
2. **mix-poker-app 統合**: AI サーバーを起動して接続
3. **対人テスト**: 実際にプレイして以下を評価
   - AI の配置は自然か？（人間から見て納得感のある手を打つか）
   - ACE の扱いは正しいか？（AA を最強ペアとして扱うか）
   - FL での配置は合理的か？
   - 明らかなミス（フォール必至の配置、意味不明なディスカード）がないか

### 成功基準
- 数値上の指標だけでなく、**定性的な評価**が重要
- 「中級者のプレイヤーとして自然な手を打つ」が最低ライン
- 明らかなルール理解の間違い（V1 の ACE バグのような）がないこと

## V2.1/V3 Future Work（V2 本線のスコープ外）

V2 では以下の改善を**意図的に含めない**。
V2 のゴールは「one-hot + MaskablePPO で V1 を超える健全なモデル」の完成であり、
アーキテクチャの大幅変更は V2 完了後に別ブランチで検証する。

### カード Embedding 層の導入（V2.1 候補）

```python
# カード表現を one-hot(54次元) から Embedding(16次元) に変更
# ネットワーク入力層にカスタム Feature Extractor を追加
policy_kwargs = dict(
    features_extractor_class=OFCFeatureExtractor,
    features_extractor_kwargs=dict(card_embedding_dim=16),
    net_arch=dict(pi=[512, 256, 128], vf=[512, 256, 128]),
)
```

**V2 本線に入れない理由**:
- ネットワーク入力構造が変わるため、V2 チェックポイントとの互換性が完全に失われる
- 導入する場合はゼロから再学習が必要（30M ステップ以上）
- V2 の MLP + one-hot でどこまで行けるかを先に確認すべき
- Embedding の効果は V2 のベースラインがあって初めて比較可能

**V2.1 検証計画**（V2 Phase 4 完了後）:
1. V2 最終モデルをベースラインとして記録
2. Embedding 版を V2 の最良パラメータ（報酬、プール、ローテーション）で 30M 学習
3. 同一ベンチマーク相手で比較
4. 有意な改善があれば 250M まで拡張

### FL ソルバー依存の解消（V3 候補）

現在の学習では FL 配置を `greedy_fl_solver`（外部ソルバー）に委任しており、
AI 自身は FL 中の配置を学習していない。

**V2 でこの問題を解消しない理由**:
- FL 配置の行動空間は通常ゲームとは桁違い（14-17 枚を 13 スロットに配置）
- 通常ゲームの学習が健全でないうちに FL 学習を追加すると問題の切り分けが困難
- V2 ではまず「通常ゲーム + FL エントリー判断」の品質を確立する

**V3 検証計画**（V2.1 完了後の長期目標）:
1. FL 専用の行動空間設計（階層的アクション or サブポリシー）
2. FL 配置の報酬設計（Stay 成功報酬 + 配置品質報酬）
3. 段階的導入：まず FL=14 枚（最も制約が厳しい）から開始
4. Greedy Solver の配置をイミテーション学習で初期化する選択肢も検討
